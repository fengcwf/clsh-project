# constitution.md — 项目宪法

> 本文件定义项目约束和不可违背的规则。所有阶段的产出必须符合此宪法。

---

## 项目约束 / Project Constraints

### C-1: 代码必须可运行

所有交付的代码必须经过测试验证，不得包含语法错误或未解决的依赖。

### C-2: 变更必须可追溯

每次文件变更必须在 TECH.md 的文件变更范围中登记，并说明变更原因。

### C-3: 接口必须向后兼容

任何公共接口的变更不得破坏现有调用方的行为。如需破坏性变更，必须通过新版本号体现。

### C-4: 不变量必须可测试

PRODUCT.md 中定义的每个不变量 (INV-n) 必须有对应的自动化测试。

### C-5: 文档与代码同步

代码变更后，相关文档 (README, TECH.md) 必须同步更新。

---

## 禁止操作 / Prohibited Actions

以下操作在任何阶段均 **绝对禁止**:

| ID | 禁止操作 | 原因 |
|----|----------|------|
| P-1 | 删除或覆盖 constitution.md 的约束条款 | 宪法不可被自身修改 |
| P-2 | 在未更新 TECH.md 的情况下修改文件结构 | 保持变更可追溯 |
| P-3 | 跳过测试直接标记任务完成 | 质量门禁不可绕过 |
| P-4 | 引入未在 TECH.md 中声明的外部依赖 | 防止依赖膨胀 |
| P-5 | 硬编码敏感信息 (密钥、密码、Token) | 安全性不可妥协 |
| P-6 | 在代码中使用 `TODO`/`FIXME` 作为最终实现 | 代码质量 |

## 实现细节规范 / Implementation Details Spec

### 编码规范 / Encoding

- 源代码文件编码: UTF-8 (无 BOM)
- 行尾符: LF (Unix 风格)
- 缩进: [根据语言选择，例如 4 空格 / 2 空格 / Tab]
- 字符串默认编码: UTF-8

### 错误处理 / Error Handling

```
错误格式: [MODULE]-[ERROR_CODE]: [描述信息]

示例:
CONFIG-001: 配置文件格式无效
PARSE-002: 输入数据解析失败
IO-003: 文件读取超时
```

- 所有错误必须包含模块标识和错误码
- 用户可见错误必须可操作 (告知如何修复)
- 内部错误必须记录完整堆栈信息
- 致命错误 (Fatal) 必须退出并返回非零退出码

### 日志规范 / Logging

- 级别: DEBUG < INFO < WARN < ERROR < FATAL
- 生产环境默认级别: INFO
- 日志格式: `[ISO-时间戳] [级别] [模块] 消息`
- 日志输出: stderr (错误), stdout (正常输出)

### 版本管理 / Versioning

- 版本格式: MAJOR.MINOR.PATCH
- MAJOR: 不兼容的 API 变更
- MINOR: 向后兼容的功能新增
- PATCH: 向后兼容的缺陷修复

## 宪法修订规则 / Amendment Rules

1. 宪法文件本身只能通过手动编辑更新
2. 新增约束不得与现有约束冲突
3. 删除约束需说明影响评估
4. 修订历史记录在本文件底部维护
