# Anti-Rationalization Patterns — 为什么 LLM 会跳过规则以及如何阻止

## 核心洞察

LLM 不是"不会"遵守规则，而是会**用看似合理的理由解释为什么可以跳过规则**。这种行为称为"理性化"（rationalization）。常规的"在 prompt 中强调规则"无法有效阻止它，因为 LLM 会重新解释规则以适应自己的行为。

---

## The 3 Reliable Anti-Deviation Strategies

### Strategy 1: Tool Interception（工具拦截）

**原理**：不依赖 LLM 自觉遵守规则，而是在工具层面强制执行。

**示例**：
```
# 传统方式（不可靠）：
prompt: "记得在每个任务中添加验证命令"
LLM: "好的，我会的" → 然后忘记

# Tool Interception（可靠）：
在 delegate 工具中强制要求 goal 字段包含 verify 命令
→ 不包含则工具报错，无法执行
```

**应用场景**：
- 任务必须有 `verify` 字段
- 产出文件必须通过格式检查
- 部署前必须有测试通过记录
- 回退必须经过特定审批步骤

**实施要点**：
1. 在工具的 input schema 中设置 `required: true`
2. 在工具执行前进行格式验证
3. 验证失败直接拒绝，不给 LLM 解释机会

### Strategy 2: Process Isolation（流程隔离）

**原理**：将关键决策点从主流程中隔离出来，由独立的（甚至不可信的）实体执行。

**示例**：
```
# 同一 Agent 自审自评（不可靠）：
LLM 写代码 → LLM review → "看起来不错"
→ rationalization: "这个实现很优雅，不需要改"

# Process Isolation（可靠）：
LLM 写代码 → 独立 reviewer（不同角色、不同上下文）→ 必须逐条对照 checklist
```

**应用场景**：
- Review 必须由独立角色执行
- 测试必须由非实现者验证
- 架构决策必须经过多角度讨论
- 安全审查必须由专门角色处理

**实施要点**：
1. 不同角色使用不同的上下文窗口
2. Reviewer 不能访问实现者的思维过程
3. 使用标准化的 checklist 而非开放式评估

### Strategy 3: Untrusted Review（不可信审查）

**原理**：假设 LLM 的产出可能存在缺陷，用另一层验证来捕获。

**示例**：
```
# LLM 自报成功（不可靠）：
subagent: "我已经完成了任务 ✓"
orchestrator: "好的，标记为完成"

# Untrusted Review（可靠）：
subagent: "任务完成"
orchestrator: 检查 diff → 检查 test output → 检查 scope compliance
→ 只有证据通过才标记完成
```

**应用场景**：
- Subagent 的"成功"必须有证据支撑
- 代码变更必须验证实际文件改动
- 测试结果必须检查原始输出而非声明
- 部署结果必须验证服务状态

**实施要点**：
1. 定义明确的"成功证据"清单
2. 验证必须是可自动化的
3. 不接受"已经做了"而要求"证明做了"

## LLM Rationalization Patterns

### Pattern 1: "这太简单了不需要"

**触发场景**：任务看起来很简单
**LLM 逻辑**：这个任务这么简单，跳过验证/文档也没关系
**真实后果**：简单任务的错误更难发现（因为没人检查）
**阻止方式**：工具强制要求，无例外

### Pattern 2: "我们之前已经做过了"

**触发场景**：在上下文中看到类似的工作
**LLM 逻辑**：这个已经做过了/验证过了
**真实后果**：假设被当作了事实
**阻止方式**：要求提供文件引用作为证据

### Pattern 3: "这样做效率更高"

**触发场景**：流程步骤看起来可以合并
**LLM 逻辑**：把 A 和 B 合在一起做更快
**真实后果**：合并后 A 和 B 都做不好
**阻止方式**：流程定义禁止合并特定步骤

### Pattern 4: "根据经验这样是对的"

**触发场景**：有类似的训练数据或案例
**LLM 逻辑**：根据我的知识，这样做是正确的
**真实后果**：训练数据可能过时或不适用当前上下文
**阻止方式**：要求基于实际代码证据而非经验

### Pattern 5: "这个约束不适用于当前情况"

**触发场景**：约束条件与直觉冲突
**LLM 逻辑**：当前情况特殊，约束不适用
**真实后果**：约束被绕过，问题出现
**阻止方式**：约束有明确的适用范围和例外条件

### Pattern 6: "我可以先做这一步再补充其他的"

**触发场景**：有多个步骤需要完成
**LLM 逻辑**：先做核心的，其他的稍后补充
**真实后果**：稍后永远不会来，其他步骤被永久跳过
**阻止方式**：工具/流程禁止部分完成

### Pattern 7: "这已经在我的上下文中了"

**触发场景**：信息在记忆中存在
**LLM 逻辑**：我已经知道这个信息，不需要从文件读取
**真实后果**：记忆可能不准确或已过时
**阻止方式**：强制从文件读取，不接受记忆

### Pattern 8: "用户可能更喜欢这个方案"

**触发场景**：有多种合理选择
**LLM 逻辑**：用户可能更喜欢这个（实际更简单）的方案
**真实后果**：绕过了正式的设计决策流程
**阻止方式**：所有选择必须经过正式 proposal 流程

## Design Principles for LLM-Resistant Workflows

### Principle 1: 将规则嵌入工具而非 prompt

```
❌ 在 prompt 中说："请在每个任务中添加验证命令"
✅ 在工具 schema 中：goal 必须包含 verify 字段
```

**原因**：Prompt 中的规则可以被"理性化"绕过，工具层面的限制不能。

### Principle 2: 不信任声明，只信任证据

```
❌ subagent: "任务完成" → 标记完成
✅ subagent: "任务完成" → 检查 diff + test output + file changes
```

**原因**：LLM 可以自信地报告虚假成功，但无法伪造文件系统级别的证据。

### Principle 3: 将验证与执行分离

```
❌ 实现者同时负责验证
✅ 实现者负责实现，验证由独立角色执行
```

**原因**：同一实体无法客观评估自己的产出。

### Principle 4: 强制中间产物，不允许隐式传递

```
❌ A → B（A 的输出直接作为 B 的输入）
✅ A → file → B（A 的输出写入文件，B 从文件读取）
```

**原因**：文件作为硬性记录，防止上下文窗口中的信息被遗漏或修改。

### Principle 5: 定义最小可接受标准，不允许"更好"

```
❌ "请写出最好的代码"
✅ "代码必须通过 lint + 测试 + 单文件 < 300 行"
```

**原因**：LLM 可能为了"更好"而超出范围，明确的最小标准防止过度。

### Principle 6: 错误必须有成本

```
❌ 出错 → 重试（无成本）
✅ 出错 → 记录 + 暂停 + 需要明确恢复策略（有成本）
```

**原因**：无成本的错误会导致 LLM 不重视错误处理。

### Principle 7: 定期刷新上下文

```
❌ 长 session 中依赖早期信息的记忆
✅ 每个 phase 开始时重新读取所有相关文件
```

**原因**：LLM 在长上下文中会丢失信息，文件作为持久化存储更可靠。

## When Human-in-the-Loop is Essential vs When Automation Suffices

### ✅ Human-in-the-Loop Essential

| 场景 | 原因 | 替代方案 |
|-----|------|---------|
| 需求确认 | 用户意图无法自动化验证 | N/A（必须人工） |
| 架构决策审批 | 涉及业务 trade-off | N/A（必须人工） |
| 安全关键代码审查 | 可能需要专业知识判断 | 人工 + 专项安全工具 |
| 生产部署批准 | 高风险操作 | 人工 + 回滚计划 |
| Constitution 修订 | 改变项目约束 | 人工 + 影响分析 |
| 跨项目影响评估 | 需要全局视野 | N/A（必须人工） |

### ✅ Automation Sufficient

| 场景 | 自动化方式 | 注意事项 |
|-----|-----------|---------|
| Lint 检查 | CI/CD + linter | 配置好的规则 |
| 测试执行 | test runner | 需要覆盖关键路径 |
| 格式检查 | validator | 模板格式 |
| 依赖安全审计 | npm audit | 定期更新 |
| 代码行数统计 | wc/脚本 | 作为辅助 |
| Diff 审计 | git diff | 作为证据收集 |

### ⚠️ 混合模式

| 场景 | 自动化部分 | 人工部分 |
|-----|-----------|---------|
| 代码 Review | 自动 lint + 测试 | 人工审查逻辑和设计 |
| 性能测试 | 自动 benchmark | 人工评估是否可接受 |
| 文档生成 | 自动生成框架 | 人工补充说明和示例 |
| 数据迁移 | 自动脚本执行 | 人工验证数据完整性 |

## Practical Examples

### Example 1: 防止跳过 Scout 阶段

**问题**：LLM 收到需求后直接写 proposal
**解决**：
1. Tool: 检查 `tech-constraints.md` 是否存在且 ≥3 条约束
2. Flow: Phase 2 入口强制读取 Phase 1 产出
3. Review: C7 检查 proposal 是否引用了 tech-constraints

### Example 2: 防止任务过大

**问题**：一个任务包含多个文件的修改
**解决**：
1. Tool: tasks.md 模板验证每个任务的 scope ≤1 文件
2. Flow: 超过 1 文件的任务自动拆分
3. Review: orchestrator 在派发前检查 scope

### Example 3: 防止 Self-Review

**问题**：实现者自己审查自己的代码
**解决**：
1. Tool: review 角色必须是不同角色
2. Flow: review 需要独立的上下文
3. Review: C7 检查 reviewer 不在 tasks.md 的 coder 角色中

### Example 4: 防止 Subagent 虚假成功

**问题**：subagent 报告成功但实际没有完成
**解决**：
1. Tool: 验证 diff 不为空
2. Flow: 检查 verification output
3. Review: orchestrator 检查实际文件变化 vs 任务描述

## Summary: The Anti-Rationalization Checklist

1. ✅ 每个规则都有对应的自动化验证
2. ✅ 关键步骤由独立角色执行
3. ✅ 成功判定基于证据而非声明
4. ✅ 中间产物必须写入文件
5. ✅ 最小可接受标准明确定义
6. ✅ 错误有明确成本
7. ✅ 定期从文件刷新上下文
8. ✅ 人工审批关键决策
9. ✅ 工具层面强制格式验证
10. ✅ 不信任"我已经知道"的声明
