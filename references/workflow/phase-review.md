# Review Mode — 规格合规审查流程

## 概述

当大佬要求 review 已有项目的代码和实现效果时，执行规格合规审查：对比项目的设计文档检查实际实现。不走完整 Phase 1-8，不创建 kanban 卡，不派发任务。

## 流程（7 步）

### Step 1: 读取项目文档

读取 `raw/projects/<项目名>/` 下的所有文档：

| 文件 | 用途 | 必读 |
|------|------|------|
| `overview.md` | 项目状态、技术栈、功能范围 | ✅ |
| `changes/<变更名>/proposal.md` | 技术方案（架构、API、前端设计） | ✅ |
| `changes/<变更名>/constitution.md` | 技术约束（规范、禁止事项） | ✅ |
| `changes/<变更名>/tasks.md` | 实现计划（任务、验收标准） | ✅ |
| `changes/<变更名>/test-log.md` | 测试记录 | ✅ |
| `changes/<变更名>/round*-feedback.md` | 历轮反馈 | ✅ |

### Step 2: 读取所有代码

逐文件阅读项目目录下的所有源文件。**不跳过任何文件。**

为什么全量阅读：
- 代码重复问题只在对比多个文件时才能发现（如同一函数在 collector.mjs 和 switch.mjs 都有实现）
- 路径不一致（如 DB_PATH 在不同文件层数不同）需要跨文件对比
- 死代码/未调用函数需要全局搜索才能识别

### Step 3: 验证运行状态

```bash
# 进程状态
pm2 list

# 依赖检查
ls node_modules/<package> && echo "OK" || echo "MISSING"

# 环境变量（脱敏）
grep -E 'KEY_PATTERN' .env | sed 's/=.*/=***/'

# API 实际调用（如果可以）
curl -s http://localhost:<port>/api/<endpoint>
```

### Step 4: 产出功能合规矩阵

按 proposal.md 中的功能列表，逐项检查实现状态：

| # | 功能 | 后端 | 前端 | 状态 |
|---|------|------|------|------|
| 1 | 功能名 | 文件名 + 关键函数 | 组件/页面 | ✅/⚠️/❌ |

状态定义：
- **✅ 完整实现** — 后端 API + 前端展示 + 数据流通
- **⚠️ 部分实现** — 后端有但前端缺，或反之
- **❌ 未实现** — 代码中找不到对应实现

### Step 5: Constitution 合规检查

逐条对比 constitution.md 中的约束 vs 实际代码：

| 约束 | 是否遵守 | 备注 |
|------|---------|------|
| 约束内容 | ✅/❌ | 具体证据（文件名、行号） |

### Step 6: 代码质量分析

按严重性分级：

| 级别 | 定义 | 示例 |
|------|------|------|
| P1 | 数据一致性/正确性 | DB_PATH 不一致、数据丢失风险 |
| P2 | 代码重复 | 同一逻辑在多处实现 |
| P3 | 架构问题 | 单例 DB 连接、全局状态、未关闭资源 |
| P4 | 死代码 | 未调用的函数、未使用的变量 |
| P5 | 性能问题 | N+1 查询、缺少缓存、阻塞操作 |
| P6 | 代码风格 | 命名不一致、缺少注释、文件过大 |

### Step 7: 汇报

汇报结构：
1. **项目进度概览** — 阶段、文件数、依赖状态
2. **功能合规矩阵** — 表格
3. **Constitution 合规检查** — 表格
4. **代码质量问题** — P1-P6 分级
5. **未实现功能清单** — 复杂度评估
6. **总结** — 完成度百分比、关键发现
7. **建议下一步** — 优先级排序

## 注意事项

- Review Mode 不创建 kanban 卡，不派发任务
- Review Mode 的产出是报告，不是修复行动
- 修复应由大佬确认后，走正常 Phase 8 反馈循环
- 如果大佬说"顺便修了"，提醒走角色分离流程
