# Review Mode — 审计维度框架

## 适用场景

当 clsh-project 以 Review Mode 启动时（"review 项目"、"代码审查"、"全面审计"），产出是**审查报告**而非代码修复。修复走 Phase 8。

## 审计维度模板（10 维度）

初始维度（代码级，通常第一轮就能覆盖）：
1. **重复造轮子** — 代码重复、冗余脚本、未使用的文件/依赖
2. **代码质量** — 错误处理一致性、性能 bug、模式统一
3. **UI 统一** — CSS 变量、组件复用、设计 token 冲突
4. **项目化管理** — 插件/模块结构、路由模式、依赖管理

深度维度（需要安全/架构视角）：
5. **安全审计** — 注入漏洞、认证绕过、CSRF、输入校验、密钥管理
6. **配置管理** — .env 处理、敏感信息、环境变量文档
7. **API 设计** — 限流、版本控制、错误格式、请求校验
8. **测试覆盖** — 自动化测试、测试框架、覆盖策略
9. **部署就绪** — 优雅关闭、健康检查、进程管理
10. **性能** — 分页、缓存、查询优化、内存管理

## Phase 1 流程（Review Mode）

1. **初始扫描** — 并行子任务覆盖维度 1-4（代码级分析）
2. **保存初始报告** — 写入项目根目录作为参考（如 `CODE-REVIEW-YYYY-MM-DD.md`）
3. **调研扩展** — 派子任务扫描维度 5-10（安全/架构视角），检查实际代码确认相关性
4. **向大佬确认范围** — 列出所有维度 + 优先级，让大佬选择审哪些
5. **⚠️ 初始报告是参考，不是执行依据** — 大佬可能扩展/缩小/重新定义范围

## 关键 Pitfall

- **Pitfall: 把初始发现当最终报告** — 第一轮扫描通常只能覆盖代码表面（重复、格式、结构）。安全/性能/配置需要深度分析。不要在初始报告后就认为"审完了"。
- **Pitfall: 假设所有维度都相关** — 有些项目没有 GraphQL（安全审计不适用）、没有 API（限流不适用）。每个维度要先检查"这个项目有没有这个问题"再深度审计。
- **Pitfall: Review Mode 直接修代码** — Review 产出报告，修复走 Phase 8。即使发现严重漏洞也不在 Review 阶段修。

## 报告结构建议

```markdown
# <项目> 代码审查报告
**审查日期**：YYYY-MM-DD
**审查范围**：<路径>
**代码规模**：<统计>

## 维度 N: <名称> <优先级图标>
### 发现
- 具体问题（文件:行号）
### 严重度
### 建议

## 整改优先级表
| 优先级 | 行动 | 预计收益 |
```
