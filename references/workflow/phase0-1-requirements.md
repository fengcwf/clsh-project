## Phase 0: 内化历史教训

**每次项目开始前必做。** 

### Step A — 快速参考
读取 `~/.hermes/skills/productivity/project-wrap-up/learnings.md`，提取与当前项目类型相关的教训，在后续 Phase 中主动应用。

### Step B — 跨项目知识复利
读取 `wiki/solutions/INDEX.md`，按当前项目的 tech/domain 匹配 `reusability: cross-project` 的条目：
1. 从大佬的需求描述中提取 tech 标签（如 vue、fastify、css）和 domain 标签（如 frontend、auth）
2. 在 INDEX.md 中查找匹配的 solutions
3. 最多注入 5 条最相关的条目摘要到本轮上下文
4. 无匹配 → 跳过（不强制）

**不产生输出文件，仅影响灵犀行为。**

## Phase 1: 需求澄清

**目标：** 理解大佬真正想要什么，发现大佬没想到的需求点。

### 规则
- **一次只问一个问题** — 不要一次抛出 5 个问题
- **多选题优先** — 比如"这个通知方式你倾向哪种？A.飞书 B.邮件 C.短信 D.都要"
- **先理解目的，再讨论细节** — 问"这个功能主要解决什么问题？"而不是"数据库用什么字段？"
- **主动提出大佬没想到的需求**
- **如果需求太大，先帮助拆分子项目"

### 🎨 Visual Companion（UI/视觉项目专用）

**触发条件：** Phase 1 涉及视觉/UI 问题（布局、交互流程、页面结构）时，在提出第一个视觉相关问题前提供。

**UI 关键词检测（自动触发）：** 以下关键词出现在需求描述中时，自动激活 Visual Companion：
- 页面、界面、UI、前端、网页、仪表板、Dashboard
- 布局、交互、响应式、移动端
- 登录页、落地页、Landing、后台管理

**模板：**
> "这个项目涉及 UI/视觉设计。我可以启动 Visual Companion 帮助你：
> - 加载 Open Design 的 152 个设计系统作为参考（Stripe/Linear/Vercel 等）
> - 生成 2-3 个设计方向的 HTML 原型
> - 沙盒预览，实时对比
> 
> 需要吗？（默认：否）"

**执行方式：**
- 大佬同意 →
  1. 从 `/opt/open-design/design-systems/` 选取 2-3 个匹配的设计系统作为参考
  2. 使用 `sketch` skill 生成 2-3 个 HTML 设计方案
  3. 启动本地 HTTP 服务（`python3 -m http.server`），在浏览器中展示
  4. 大佬在浏览器中查看并反馈，灵犀根据反馈调整方案
  5. 完成后关闭服务
- 大佬不同意 → 跳过，继续需求澄清

**⚠️ 注意：** 仅用于视觉/UI 讨论。纯后端/API 项目不需要。

### 🔄 Phase 中断恢复（Restart Protocol）

如果 session 中断后重启（如新会话、网关重启等），**不要从头开始 Phase 1**，而是：

1. `session_search` 搜索最近 3 个 session，关键词用项目名
2. 读取历史 session 的 summary，恢复上下文
3. 检查 `raw/projects/<项目名>/` 是否已有文档产出
4. 从上次中断的 Phase 继续，**不要重复已完成的工作**
5. 向大佬确认恢复的上下文是否正确，再继续提问

### 🔍 调研循环（每个决策点：调研→提问→确认）

**替代旧的"调研前置"模式。** 不是一开始调研一次就结束，而是每个决策点都走 `调研→提问→确认` 微循环，调研范围逐轮收敛。

#### 调研三层模型

| 层级 | 范围 | 何时触发 | 工具 |
|------|------|---------|------|
| **L0 内部** | 代码库、wiki、raw/projects、现有配置、历史教训 | **每个决策点必做** | read_file、search_files、session_search |
| **L1 外部** | GitHub 同类项目、官方文档、竞品实现 | 技术选型/方案不确定时 | web_search、web_extract |
| **L2 深度** | 技术博客、社区讨论、论文/研报 | 架构级决策/L1 结论不够时 | web_search、web_extract |

#### 决策点调研流程

```
大佬提出需求
  ↓
L0 调研：扫描代码+wiki+配置 → 了解现状
  ↓ 提问（带推荐答案）→ 大佬回答
  ↓
回答触发新问题？
  ├─ 是 → L0 细化调研（基于回答缩小范围）→ 提问 → ...
  ├─ 涉及技术选型 → L1 外部调研（GitHub/文档/竞品）→ 带结果提问 → ...
  └─ 架构级决策 → L2 深度调研（文章/讨论）→ 带结果提问 → ...
  ↓
所有分支解决 → 进入 Phase 2
```

#### 规则

1. **L0 必做** — 每个决策点先查内部再问，不空问
2. **L1/L2 按需触发** — 不是每个问题都需要外部调研。判断标准：大佬说"不确定"/"帮我调研"/"有什么方案" → 触发 L1
3. **调研结果附在提问中** — 不是调研完自己存着，而是带着结果问："调研了 3 个方案，推荐 B，因为..."
4. **每轮调研缩小范围** — 第 1 轮了解全局，第 2 轮聚焦某个分支，第 3 轮深入具体技术细节
5. **大佬可以随时叫停** — "够了，不用再调研了" → 停止外部调研，基于已有信息继续

#### 机械确认门禁（每个决策点必过）

**原则：** 前面的调研和提问可以靠 LLM 质量，但流程推进的控制权在机械判断 — 没收到正确确认码就不往下走。

**流程：**
```
L0 调研 → 提问(带推荐) → 灵犀展示总结 + 随机确认码
  ↓
[机械门禁] 等待码
  ├─ 大佬回复码（如 "x7k2"） → 码匹配 → 放行 → 下一个决策点
  ├─ 大佬回复新问题/修改意见 → 不放行 → 继续当前决策点
  └─ 大佬没回复 → 死等
```

**灵犀展示格式（每个决策点必须用这个格式结尾）：**
```
━━ 决策点 N/M: <标题> ━━

[调研结果摘要]

推荐：<答案>（基于 <依据>）

选项：
A. ...
B. ...
C. ...

🔑 `x7k2`
📋 复制上面的码回复即可
```

**规则：**
1. 每个决策点结束时，灵犀必须用以下脚本生成 4 位随机确认码（**禁止 LLM 自行编造**）：
   ```python
   python3 -c "import secrets,string; print(''.join(secrets.choice(string.ascii_lowercase+string.digits) for _ in range(4)))"
   ```
2. 码附在展示末尾，格式为单独一行 `` 🔑 `xxxx` ``（反引号包裹，方便复制）
3. 大佬直接回复码即放行（不需要"确认"前缀）
4. 码不匹配 → 不放行，提示"请输入正确确认码"
5. 大佬回复其他内容（新问题、修改意见）→ 视为对当前决策点的补充，继续讨论，讨论完再生成新码
6. **所有决策点都确认后** → 灵犀展示决策树总结 + Phase 1 确认模板（内嵌于 SKILL.md）→ 大佬回复码 → 进入 Phase 2
7. 灵犀不得跳过门禁直接进入下一个决策点或 Phase 2
8. 灵犀不得自行编造确认结果

#### 决策树追踪

Phase 1 结束时，conversation.md 中必须包含决策树：

```markdown
## 决策树
- [x] 目标：内容发布引擎 → 已确认
- [x] 渠道：Hugo + 微信 + 小红书 → 已确认
- [x] 微信方案：云主机代理获取 token → L1 调研后确认
- [ ] 图片生成：AI 生成 vs 手动上传 → 待定
  - 依赖：目标确认 ✅
```

每个节点标记：状态（已确认/待定/阻塞）、来源（L0/L1/L2 调研 or 大佬直接决定）。

#### L1 外部调研规范

当触发 L1 调研时：

1. **搜 GitHub** — 搜同类项目，关注 star 数、最近更新、架构选型
2. **读官方文档** — API 限制、配置要求、已知坑
3. **看竞品** — 别人怎么做的，有什么教训

输出格式：
```markdown
## L1 调研：<方向>
- GitHub 项目：[名称](url) — star/最近更新/核心特点
- 关键发现：[3-5 条，聚焦与本项目相关的]
- 推荐方案：[X]，因为 [理由]
- 风险/限制：[已知坑]
```

### 🔬 代码交叉验证（Phase 1 必做）

**来源：** Matt Pocock /grill-with-docs 的 "Cross-reference with code"

在需求澄清过程中，当大佬描述现有系统/功能/行为时：
1. **先查代码验证** — 不盲目接受描述，代码是事实真相
2. **发现矛盾立即指出** — "你说 X 功能是这样工作的，但代码显示的是 Y — 哪个是对的？"
3. **记录验证结果** — 在 conversation.md 中标注哪些描述已验证、哪些有差异

**适用场景：**
- 大佬说"现有的系统是这样做的" → 必须查代码
- 大佬说"这个功能应该没问题" → 检查实际状态
- 大佬描述 API 行为 → 对照代码确认

**⛔ 不适用：** 全新项目（无代码可查）、纯需求讨论（不涉及现有行为）

### 📖 领域语言积累（CONTEXT.md）

**来源：** Matt Pocock /grill-with-docs 的 CONTEXT-FORMAT.md

Phase 1 需求澄清过程中，当出现项目特有术语时，同步记录到 context.md：

1. **识别术语** — 大佬使用的项目特有词汇（不是通用编程概念）
2. **当场定义** — 问清含义，一两句话定义"是什么"
3. **列出 Avoid** — 同义词/容易混淆的词
4. **写入 context.md** — 不攒到最后，边讨论边记录

**文件位置：** `raw/projects/<项目名>/source-of-truth/context.md`
**模板：** `references/templates/context-template.md`

**与其他文档的关系：**
- conversation.md 记录 Q&A，引用 context.md 术语
- proposal.md 使用 context.md 术语保持一致
- constitution.md 不重复 context.md 内容

### 提问模板（按顺序探索）

1. **目的和用户** — "这个功能主要是给谁用的？解决什么问题？"
2. **核心功能** — "核心要做什么？最小可用版本包含哪些功能？"
3. **输入输出** — "用户输入什么？系统输出什么？"
4. **边界条件** — "异常情况怎么处理？"
5. **非功能需求** — "性能要求？安全要求？需要兼容什么环境？"
6. **优先级** — "如果时间紧，哪些功能可以后做？"

### 需求文档格式

**文件位置：** `raw/projects/<项目名>/changes/YYYY-MM-DD-<变更名>/conversation.md`

## ⛔ Common Pitfalls（Phase 0-1 高频）

> 从 pitfalls/common.md 提取的 Phase 0-1 高频教训，避免额外查文件。

### #58/#83 查找优先级 — 先查 skill/memory，最后才问用户

**规则：** 需要环境配置、凭据、端口、路径等信息时，查找优先级 = (1) 查相关 skill → (2) 查 memory → (3) 查 wiki/Obsidian → (4) 最后才问用户。

**反例：** 代理 `192.168.0.41:7890` 已记录在 `proxy-workarounds` skill 中，但灵犀没查就问用户。大佬纠正："代理信息应该在 clsh-project 里有记载，为什么需要问我"。

### #56 Phase 1 需求澄清顺序

**规则：** 先理解目的（解决什么问题），再讨论细节（数据库字段、API 路径）。

**反例：** 一上来就问"数据库用什么字段？"→ 大佬还没确认要做什么功能。

### 代码交叉验证（Phase 1 必做）

**规则：** 大佬描述现有系统行为时，必须查代码验证。发现矛盾时以代码为准，向大佬确认。

**反例：** 大佬说"现有的搜索功能是这样做的" → 灵犀直接接受 → Phase 6 发现代码和描述不一致 → 返工。
