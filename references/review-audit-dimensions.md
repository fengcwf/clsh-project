# Review Mode 审计维度框架

## 适用场景

Review 项目（代码审查、架构审查、安全审查等）使用 clsh-project 的 Phase 1-7 流程，但产出是审查报告而非代码修改。

## 审计维度模板（10 维度）

### 代码级维度（第一批）

| # | 维度 | 审计项 |
|---|------|--------|
| 1 | 重复造轮子 | 临时脚本、共享工具使用率、DB 连接模式、fetch 封装复用、孤儿文件 |
| 2 | 代码质量 | 错误处理一致性、性能反模式、注释覆盖率、异常处理完整性 |
| 3 | UI 统一 | CSS 变量使用率、设计 token 一致性、共享组件库、view 共享模块导入率 |
| 4 | 项目化管理 | 插件结构一致性、路由模式、依赖声明、错误处理中间件 |

### 安全/架构维度（第二批）

| # | 维度 | 审计项 |
|---|------|--------|
| 5 | 安全审计 | 注入漏洞（SQL/GraphQL/XSS）、认证/授权、CSRF、输入校验、密钥管理 |
| 6 | 配置管理 | .env 处理安全性、环境变量文档、敏感信息保护、硬编码路径 |
| 7 | API 设计 | 限流/防暴力破解、错误响应格式、请求校验（schema）、版本控制 |
| 8 | 测试覆盖 | 自动化测试存在性、测试框架配置、关键路径覆盖 |
| 9 | 部署就绪 | 优雅关闭、健康检查、进程管理配置、日志配置 |
| 10 | 性能 | 分页、查询缓存、连接管理、内存管理 |

## Phase 1 流程

1. **初始扫描**: 并行子任务覆盖代码级维度（①-④），产出初始报告
2. **保存初始报告**: 写入项目根目录（如 `CODE-REVIEW-YYYY-MM-DD.md`），作为参考
3. **调研扩展**: 扫描安全/架构维度（⑤-⑩），补充新发现
4. **向大佬确认范围**: 初始报告是参考，不是执行依据；确认哪些维度纳入

## 执行方案选择

| 方案 | 并行度 | 适用场景 |
|------|--------|----------|
| A: 全并行 | N 个子任务各覆盖 1 个维度 | 最快，维度间交叉分析弱 |
| B: 分批并行 | 按相关性分 2-3 批 | 平衡速度和深度 |
| C: 两阶段 | 阶段 A 表面扫描 + 阶段 B 深度分析 | 最全面，有优先级排序 |

## 单维度报告模板

```markdown
## 维度 N: <名称> | 优先级: 🔴/🟡/🟢

### 审计标准
- 该维度要检查的具体条目

### 发现
| # | 严重度 | 位置 | 问题描述 | 证据 |
|---|--------|------|---------|------|
| 1 | 🔴/🟡/🟢 | file:line | 具体描述 | 代码片段/命令输出 |

### 统计
- 🔴 严重: N 个
- 🟡 中等: N 个
- 🟢 轻微: N 个

### 修复建议
- 按优先级排列的具体行动项
```

## 最终报告结构

```
1. 执行摘要（1 页）
   - 项目概况（技术栈、规模、子项目清单）
   - 总体评分（10 维度雷达图描述）
   - Top 5 必修问题
   - Top 5 建议改进

2. 维度详情（10 个章节）

3. 交叉分析
   - 维度间关联问题
   - 根因归类（架构缺陷 vs 历史债务 vs 缺失规范）

4. 整改路线图
   - P0: 立即修复（安全风险）
   - P1: 短期改进（1-2 周）
   - P2: 中期优化（1 月）
   - P3: 长期规划

5. 附录
```

## 严重度定义

| 级别 | 含义 | 举例 |
|------|------|------|
| 🔴 严重 | 安全漏洞、数据丢失风险、生产事故隐患 | SQL 注入、明文密钥、无认证端点 |
| 🟡 中等 | 代码质量、可维护性、一致性问题 | 重复代码、错误处理不统一、无测试 |
| 🟢 轻微 | 改进建议、最佳实践偏差 | 缺少注释、命名不规范、文档缺失 |

## Pitfall

### Pitfall #138: Review 项目范围必须含根目录
**问题**: 审查范围只覆盖 `src/`，遗漏根目录脚本、config/、scripts/、ecosystem.config.cjs 等。
**根因**: 下意识把 "源码" 等同于 "src/"，忽略根目录也有重要代码。
**解法**: Review 项目的审查范围默认为项目根目录全部文件（排除 node_modules/、.git/、data/logs/ 等运行产物）。

### Pitfall #139: 初始报告是参考不是执行依据
**问题**: 初始扫描报告直接作为最终审查结论，跳过深度调研。
**根因**: 初始报告看起来很全面，误以为已完成。
**解法**: 初始报告保存为参考 → 调研扩展维度 → 向大佬确认范围 → 范围确认后才开始正式审查。

### Pitfall #140: Phase 4 机械检查不适用于 Review 项目
**问题**: `phase4-mechanical-check.py` 对 Review 项目报 FAIL（期望 `changes/*/conversation.md` 结构）。
**根因**: 脚本针对开发项目设计，Review 项目用扁平文档结构。
**解法**: Review 项目跳过 G2 脚本检查，改用人工确认码门禁。
