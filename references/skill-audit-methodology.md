# clsh-project Skill 审计方法论

> 来源：2026-06-09 审查报告 vs 实际状态的 5 处偏差教训。
> 审计 skill 时必须按此清单逐项验证，不能凭中间状态汇报。

---

## 偏差模式（已验证）

### 模式 1: 报告基于中间状态

**场景：** 审查过程中编辑了文件，报告采集时用的是编辑前的快照，实际磁盘已是编辑后的状态。

**反例：** SKILL.md 报告写 380 行/v5.47.0，实际磁盘 391 行/v5.48.0（git 同步已升级）。

**铁律：** 报告中所有行数/版本号必须在**最终写入前**重新 `wc -l` + `grep version` 验证。

### 模式 2: 两份文件内容不同误判为新旧版本

**场景：** 同名文件存在于 raw 和 skill-local，内容不同。审查时假设一个是另一个的更新版，实际是两个独立变体。

**反例：** phase8-feedback-template.md — skill-local（"5 步流程"，引用 v5.29.0）和 raw（"反馈循环模板"，含 C0 角色分离）内容完全不同。

**铁律：** 遇到同名文件不同内容时，必须 `diff` 比较内容差异，不能只看修改时间判断权威性。选择标准：内容更完整、更符合当前架构规则的版本。

### 模式 3: 删除操作只完成一处

**场景：** 文件存在于 raw + skill-local 两处，只删了其中一处。

**反例：** verify-v530.sh 只删了 skill-local，raw 还在。

**铁律：** 删除操作必须 `find` 两处验证全部清除。

### 模式 4: 截断根因未确认就归因

**场景：** 文件被截断，报告中给出的根因是猜测而非验证。

**反例：** common.md 从 433 行截断为 25 行。报告说"编辑操作覆盖"，但具体哪个操作不清楚。

**铁律：** 根因不确定时写"根因待定"，不写确定性描述。

### 模式 5: 行数偏差未重新验证

**场景：** 审查过程中文件行数变化（编辑/同步），报告用的是旧数据。

**反例：** SKILL.md 报告写 380 行（超限 30），实际 391 行（超限 41）。

**铁律：** 行数数据必须在报告写入前最后一次 `wc -l` 获取。

## 审计清单（审查 skill 时必跑）

```bash
# 1. 版本一致性
grep '^version:' SKILL.md

# 2. 行数验证
wc -l SKILL.md

# 3. 双位置文件同步验证
diff skill-local/file raw/file

# 4. 格式残留扫描
grep -rn '旧格式模式' raw/ skill-local/

# 5. git 状态 vs 磁盘状态
cd skill-local && git diff HEAD -- . | head -20

# 6. 文件存在性验证
find skill-local raw -name 'filename' 2>/dev/null
```

## SKILL.md 瘦身技术

当 SKILL.md 超过 350 行限制时，按优先级迁移：

| 内容类型 | 迁移目标 | 节省行数 |
|---------|---------|---------|
| Pitfall 完整表（场景+根因+铁律） | common.md | 每条约 3 行 |
| 近期 Pitfalls 索引（15+ 行表格） | 一行指针到 common.md | ~20 行 |
| Phase 8 Pitfalls 表 | common.md（已有对应条目） | ~10 行 |
| Anti-Rationalization 表 | common.md 或独立参考文件 | ~15 行 |
| 旧铁律降级表 | 一行摘要 + 指针 | ~15 行 |

**操作模式：** 详细表→一行摘要+`详见 common.md`。不丢失信息（common.md 有完整内容），只减少 SKILL.md 的行数。
