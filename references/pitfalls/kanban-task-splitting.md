# Kanban 任务拆分教训（2026-06-08）

## 问题

组合任务（如"P0: 修 3 个 bug"）在 kanban 执行时超时（迭代预算 90/90 耗尽）。

## 根因

一个任务包含多个独立修改 → worker 需要：
1. 读取多个 bug 的上下文（每个 bug 的 spec + 相关代码）
2. 逐个修改代码
3. 逐个验证

90 次迭代不够覆盖 3+ 个独立修改的完整 cycle。

## 铁律

**一个 kanban 卡只修一个问题。**

拆分原则：
- 每个任务 ≤ 1 个文件修改
- 每个任务 ≤ 30 行代码改动
- 超时 = 拆分信号，不是重试信号

## 正确做法

```
❌ 错误：一个大任务
t_abc123: "P0: 右键菜单 + 下拉遮挡 + 毛玻璃背景"  → 超时

✅ 正确：拆成独立小任务
t_001: "右键菜单修复"（1 文件，~10 行）
t_002: "下拉遮挡修复"（1 文件，~5 行）
t_003: "毛玻璃背景"（1 文件，~15 行）
```

## 与超时重试的区别

kanban 的自动重试（max_retries=2）适用于：
- 偶发性失败（网络超时、进程被 kill）
- 迭代预算刚好不够（差 1-2 次）

不适用于：
- 任务本身太大（迭代预算根本不够）→ 拆分
- 需求不清晰（worker 反复猜测）→ 补充 spec

## 触发信号

当看到以下日志时，立即拆分：
```
Iteration budget exhausted (90/90) — task could not complete
```

不要等第 3 次超时才拆分。第 1 次超时就应分析是否需要拆分。
