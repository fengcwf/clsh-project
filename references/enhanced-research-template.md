# 调研模板增强方案 — 借鉴 Phoenix + Kiro

> 来源：2026-06-10 E2E 测试复盘，发现 11/15 问题点因调研模板缺维度而遗漏

## 问题根因

现有调研模板（Phase 0-1）只覆盖：
- 功能实现（怎么做）
- 技术选型（用什么）
- 质量规格（泛泛的安全/性能/可靠性/可观测性）

**缺失维度：**
- 安全威胁分析（攻击面、威胁建模）
- 数据隐私合规（匿名化、删除权、PIPL 条款）
- 用户体验异常场景（断网、加载失败、操作中断）
- 权限隔离规则（角色间数据边界）
- 可测试性（验收标准可直接转测试用例）

## 增强方案

### 新增维度 1：安全威胁分析（借鉴 Phoenix）

```markdown
## 安全威胁分析

### 攻击面识别
| 功能 | 攻击面 | 威胁等级 | 缓解措施 |
|------|--------|---------|---------|
| [功能名] | [可能的攻击方式] | 高/中/低 | [具体防护措施] |

### 数据敏感性分级
| 数据类型 | 敏感等级 | 存储方式 | 加密要求 |
|---------|---------|---------|---------|
| [数据名] | 高/中/低 | [存储方案] | [加密方式] |
```

### 新增维度 2：数据隐私合规

```markdown
## 数据隐私合规

### 适用法规
- [ ] 个保法（中国）
- [ ] GDPR（欧盟）
- [ ] PIPL（中国个人信息保护法）
- [ ] 其他：___

### 合规要求
| 法规条款 | 要求 | 实现方式 |
|---------|------|---------|
| 数据删除权 | 用户可请求删除所有个人数据 | 提供"删除我的数据" API |
| 数据匿名化 | 分析报表中姓名脱敏 | ID 关联，不直接显示姓名 |
| 数据导出 | 用户可导出自己的数据 | 提供数据导出 API |
| 处理目的说明 | 告知用户数据用途 | 隐私政策页面 |
```

### 新增维度 3：用户体验异常场景

```markdown
## 用户体验异常场景

| 场景 | 当前行为 | 期望行为 | 优先级 |
|------|---------|---------|--------|
| 网络断开 | 无提示 | 自动重连+离线提示 | P1 |
| 视频加载失败 | 无处理 | 重试机制+错误提示 | P1 |
| 上传中断 | 无处理 | 断点续传 | P1 |
| 页面刷新 | 状态丢失 | 恢复到刷新前状态 | P2 |
| 并发操作 | 数据冲突 | 乐观锁+冲突提示 | P2 |
```

### 新增维度 4：权限隔离规则

```markdown
## 权限隔离规则

### 角色数据边界
| 角色 | 可访问数据 | 不可访问数据 | 验证方式 |
|------|-----------|-------------|---------|
| 学生 | 自己的作业/成绩/进度 | 他人的作业/成绩 | userId 匹配检查 |
| 教师 | 自己课程的所有数据 | 其他课程的数据 | teacherId 匹配检查 |
| 管理员 | 所有数据 | — | RBAC 角色检查 |

### 权限验证矩阵
| 操作 | 学生 | 教师 | 管理员 | 验证逻辑 |
|------|------|------|--------|---------|
| 查看作业 | ✅ 自己的 | ✅ 自己课程的 | ✅ 所有 | userId/teacherId 匹配 |
| 批改作业 | ❌ | ✅ 自己课程的 | ✅ 所有 | teacherId + courseId 匹配 |
| 删除课程 | ❌ | ❌ | ✅ | RBAC 角色检查 |
```

### 新增维度 5：可测试性验收标准（借鉴 Kiro）

```markdown
## 可测试性验收标准

### GIVEN...WHEN...THEN 格式

**功能：用户登录**
- GIVEN 用户已注册 WHEN 输入正确邮箱和密码 THEN 返回 JWT token + 用户信息
- GIVEN 用户已注册 WHEN 输入错误密码 THEN 返回 401 UNAUTHORIZED
- GIVEN 用户未注册 WHEN 尝试登录 THEN 返回 401 UNAUTHORIZED
- GIVEN 用户已登录 WHEN token 过期 THEN 返回 401 + 提示重新登录

**功能：作业提交**
- GIVEN 学生已登录 WHEN 提交作业 THEN 创建 submission 记录
- GIVEN 学生已提交 WHEN 再次提交 THEN 返回 409 CONFLICT
- GIVEN 教师是课程所有者 WHEN 批改作业 THEN 更新 score+feedback
- GIVEN 教师不是课程所有者 WHEN 批改作业 THEN 返回 403 FORBIDDEN
```

## 与现有模板的集成方式

### 方案 A：扩展 conversation.md 的"调研发现"节

在现有 5 个维度基础上增加：
- 安全威胁分析（Phoenix）
- 数据隐私合规
- 用户体验异常场景
- 权限隔离规则

### 方案 B：新增独立文件 threat-model.md

在 `changes/<日期>-<描述>/` 目录下新增：
- `threat-model.md` — 威胁建模+数据隐私+权限隔离
- `acceptance-criteria.md` — GIVEN...WHEN...THEN 格式验收标准

### 方案 C：扩展 constitution.md 的"质量规格"节

在现有 4 个维度（安全/性能/可靠性/可观测性）基础上增加：
- 权限隔离规则
- 数据隐私规则
- 内容安全规则
- 防作弊规则

**推荐方案 A + C 组合：** conversation.md 调研时覆盖所有维度，constitution.md 约束时细化具体规则。

## 使用方式

Phase 0-1 调研时，按以下清单逐项检查：

```
调研完整性检查：
- [ ] 功能实现（怎么做）
- [ ] 技术选型（用什么）
- [ ] 质量规格（性能/可靠性）
- [ ] 安全威胁分析（攻击面+缓解）  ← 新增
- [ ] 数据隐私合规（法规+实现）  ← 新增
- [ ] 用户体验异常场景  ← 新增
- [ ] 权限隔离规则  ← 新增
- [ ] 可测试性验收标准  ← 新增
```
