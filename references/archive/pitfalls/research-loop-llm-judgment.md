## 调研循环 L1 触发的 LLM 判断力问题（2026-06-05 分析发现）

**问题：** Phase 1 调研循环的 L1 触发条件"技术选型/方案不确定时"是 LLM 主观判断，不同 session 可能做出不同决定。这违反了 **LLM 能力无关性原则**（SKILL.md §🛡️）。

**对比 Superpowers：** Superpowers brainstorming 不做外部调研（只查项目内部文件/文档/commit），所以不存在"调研范围划定"问题。如果问题需要外部知识，直接问用户。这虽然调研深度有限，但流程完全可控——没有 LLM 判断"要不要调研"的灰色地带。

**正确做法：** L1 触发用硬编码条件（机械判断），不用 LLM 判断。

推荐的硬编码触发条件——需求涉及以下关键词时强制触发 L1：
1. 第三方 API / SDK / 外部服务
2. 项目中没有用过的新技术栈
3. 竞品对比需求

否则只做 L0（内部调研）。

**同理：** 调研循环终止条件也需要硬编码（如最多 5 轮），不能靠 LLM 判断"够了"或"所有分支已解决"。

**与 LLM 能力无关性原则的关系：**

| 类型 | 谁判断 | 用于 | 调研循环中的对应 |
|------|--------|------|----------------|
| 机械判断 | 代码/硬编码 | 门禁检查、状态流转 | L1 触发条件、轮次上限 |
| 大佬判断 | clsh | 方案选择、方向变更 | "够了不用再调研" |
| LLM 判断 | 灵犀 | 内容生成、格式化 | ❌ 不应用于"要不要 L1" |

**教训：** clsh-project 加了外部调研层（L0/L1/L2），但没有配套的机械控制机制。这是"功能设计时没考虑执行可控性"的典型案例。
