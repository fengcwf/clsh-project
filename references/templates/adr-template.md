# {决策标题}

{1-3 句话：上下文、决策、原因。}

## 条件（满足全部 3 条才创建此 ADR）

1. **难逆转** — 改变这个决策的代价大（时间/重构/数据迁移）
2. **令人意外** — 未来读者会疑惑"为什么这样做？"
3. **有真实取舍** — 确实有其他可行方案，不是"显而易见的选择"

**如果任一条件不满足 → 不创建 ADR。** 简单决策记录在 proposal.md 或 retrospective.md 即可。

### 后续影响

## 编号规则

ADR 存放在 `raw/projects/<项目名>/docs/adr/` 目录，按顺序编号：
- `0001-slug.md`
- `0002-slug.md`

扫描目录找到最大编号，+1。

## 什么值得记 ADR

| 类型 | 示例 |
|------|------|
| 架构形态 | "用单体不用微服务"、"事件溯源" |
| 集成模式 | "A 和 B 通过事件通信，不用同步 HTTP" |
| 带锁定的技术选型 | 数据库、消息总线、部署目标（不是每个库） |
| 边界和范围 | "X 数据由 A 模块拥有，其他模块只通过 ID 引用" |
| 偏离常规的决定 | "手动 SQL 不用 ORM，因为 X" |
| 代码中看不到的约束 | "不能用 AWS 因为合规要求" |
| 非显而易见的拒绝 | "考虑过 GraphQL 但选了 REST，因为..." |
