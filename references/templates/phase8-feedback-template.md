# Phase 8 反馈循环模板

**大佬测试反馈问题后，必须按此模板执行。跳过任何步骤 = 流程违规。**

---

## Step 1: 记录反馈（灵犀做）

```
Phase 8 反馈记录：

反馈时间：YYYY-MM-DD HH:MM
反馈来源：大佬测试 / 浏览器截图 / API 测试

问题清单：
1. [ ] 问题描述：_____
   现象：_____
   复现步骤：_____
2. [ ] 问题描述：_____
   现象：_____
   复现步骤：_____
```

**规则：**
- 只记录现象，不分析原因（分析是 worker 的活）
- 不描述"可能是 XXX 导致的" — 只描述"我看到了什么"

---

## Step 2: 写 Bugfix Spec（灵犀做）

```
Bugfix Spec：

文件路径：raw/projects/<项目>/changes/<变更名>/bugfix-spec.md

内容：
## 问题 1
- 现象：_____
- 相关文件：_____
- 验收标准：_____

## 问题 2
- 现象：_____
- 相关文件：_____
- 验收标准：_____
```

**规则：**
- spec 写到文件里，不写在 task body 里（Context File Pattern，铁律 #13）
- 每个问题必须有验收标准（可执行的命令或条件）
- 绝对路径（不写相对路径）

---

## Step 3: 派 Coder 修复（顺序派发）

```
Coder 派发（顺序，不可并行）：

当前共 [N] 个修复任务：

### 问题 1 修复 | 文件：_____ | 预计：___min
- [ ] 步骤 1: _____
- [ ] 步骤 2: _____

### 问题 2 修复 | 文件：_____ | 预计：___min
- [ ] 步骤 1: _____
- [ ] 步骤 2: _____

派发规则：
- 每次只派发 1 个 coder 任务
- task body 引用 bugfix spec 绝对路径
- 等上一个完成后（kanban show 状态 = done）再派发下一个
- 完成后打勾 [x]

---
[CODE]
---
```

**规则：**
- task body 只传：Brain Dump + 问题现象 + 文件路径 + Trust Levels + Pattern to Follow + 验收标准（按 Context Engineering 五层架构）
- worker 自己读 spec + 读代码 + 分析根因 + 修复
- 灵犀不做代码推理（Way C 铁律）

---

## Step 4: 派 Tester 验证（每个 coder 完成后）

```
Tester 验证：

### 验证问题 1 修复
- [ ] 子步骤 a: _____（命令/操作）
- [ ] 子步骤 b: _____（预期结果）
- [ ] 子步骤 c: _____（截图/日志）

### 验证问题 2 修复
- [ ] 子步骤 a: _____（命令/操作）
- [ ] 子步骤 b: _____（预期结果）

派发规则：
- 每个 tester 卡可包含多个子步骤（同一功能的多个验证点）
- tester 必须实际运行命令/浏览器截图，不能只读代码
- UI 项目必须包含浏览器截图子步骤
- 验证通过后打勾 [x]
```

---

## Step 5: 汇报大佬（所有验证通过后）

```
Phase 8 修复完成：

修复结果：
- [x] 问题 1：_____（修复方式：_____）
- [x] 问题 2：_____（修复方式：_____）

验证结果：
- [x] tester 1：PASS
- [x] tester 2：PASS

请大佬再次测试确认。
```

---

## 使用规则

1. **Step 1-2 灵犀做**（记录 + 写 spec）— 不超过 2 分钟
2. **Step 3-4 派 worker**（coder + tester）— 灵犀不碰代码
3. **每个 `[CODE]`** — 填充时必须生成确认码
4. **不分析根因** — 灵犀只记录现象，worker 分析根因
5. **不自己修** — 即使"很快能修好"也必须派 worker（铁律 #4）
6. **确认码格式统一** — 码独占一行，前后无 emoji/说明文字，方便复制
