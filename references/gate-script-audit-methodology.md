# Gate 脚本审计方法论

## 4 轮模型循环审查模式

基于 clsh-darwin 的模型交替机制，对 clsh-project 的 gate 脚本和模板做系统性审查。

### 审查轮次配置

| 轮次 | 角色 | 模型 | 聚焦 |
|------|------|------|------|
| R1 | 正面审查 | 默认 | 结构完整性、修复验证 |
| R2 | 对抗审查 | 备用（DeepSeek） | 漏洞挖掘、矛盾发现 |
| R3 | 综合审查 | 默认 | 安全+可行性+改进 |
| R4 | 最终综合 | 备用（DeepSeek） | 优先级重新评估、总结 |

### 审查维度清单

1. **流程完整性** — Phase 入口/出口条件、遗漏检查
2. **门禁覆盖率** — 每个 Phase 是否有 gate 脚本
3. **模板与门禁匹配** — gate 脚本关键词是否匹配模板实际内容
4. **模板质量** — 引导性、缺失字段、TODO 注释处理
5. **脚本健壮性** — regex 过宽/过严、edge case、错误处理
6. **一致性** — 文件名/路径/规则在 SKILL.md、模板、脚本间的一致性
7. **安全性** — HMAC、确认码 entropy/TTL、路径遍历
8. **可行性** — 实际执行时间、上下文窗口压力

## Critical 降级模式（重要教训）

**R2（对抗审查）发现的"Critical"在 R3/R4 中往往被降级。** 这是正常且有价值的审查模式。

### 降级案例（v6.2.0→v6.3.0 实测）

| 问题 | R2 评级 | R3/R4 评级 | 降级原因 |
|------|---------|-----------|---------|
| verify_marker() 无调用者 | Critical | MEDIUM/P3 | 死代码无安全影响，API surface 设计 |
| GATE_SECRET 不验证 | Critical | LOW/P3 | defense-in-depth，非安全漏洞 |
| Gate 间无状态链 | Critical | MEDIUM/P2 | 文件依赖+SKILL.md 已间接保护 |

### 降级判断标准

问自己：**在实际使用场景中（LLM 自觉执行流程），这个问题会导致什么后果？**

- 如果 LLM 会遵守规则 → 问题被缓解，降级
- 如果 LLM 可以无意中绕过 → 保持评级
- 如果 LLM 必须故意违反才能触发 → 降级到 P3

### R2 对抗审查的价值

虽然"Critical"被降级，但 R2 仍然有价值：
1. 发现了 R1 未发现的 Phase 7 模板遗漏（真正需要修复）
2. 暴露了代码质量问题（dead code、API 不一致）
3. 提供了 R3/R4 重新评估的素材

## 高频发现清单

### 门禁脚本常见问题

| 问题 | 频率 | 检查方法 |
|------|------|---------|
| 关键词过宽（`design`、`scope` 匹配任何文档） | 高 | grep 每个关键词在所有文档中的出现次数 |
| MIN_NONBLANK_LINES 过低（<10 行几乎无意义） | 高 | 对比模板默认行数 |
| regex 不支持中文 | 高 | 用中文 markdown 表格测试 |
| 目录存在性检查缺失 | 中 | 传入不存在的目录测试 |
| placeholder 不覆盖中文占位符 | 中 | 检查"待定""待补充"等 |
| PASS/FAIL fallback 误匹配 | 中 | 检查否定句（"no FAIL expected"） |

### 模板常见问题

| 问题 | 频率 | 检查方法 |
|------|------|---------|
| Phase 编号与 SKILL.md 不匹配 | 高 | 逐 Phase 对比 |
| 里程碑表覆盖不全 | 高 | 检查 Phase 数量 |
| 缺少 [CODE] 占位符 | 中 | 检查每个确认模板 |
| Task 格式与 gate 脚本不匹配 | 中 | 用模板内容跑 gate 脚本 |

### 安全机制常见问题

| 问题 | 频率 | 实际风险 |
|------|------|---------|
| HMAC 签名但不验证 | 高 | LOW（死代码） |
| 确认码无 TTL | 高 | MEDIUM（旧码可复用） |
| HMAC 密钥可预测 | 中 | LOW（单机场景足够） |
| verify 函数无调用者 | 中 | LOW（API surface） |
